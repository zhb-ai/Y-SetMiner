## 一、销售分析 (6个报表)

### 1.1 销售业绩报表 ⭐⭐⭐
**业务价值**: 监控销售业绩，支持经营决策
**MV依赖**: MV_SALES_DETAIL（明细钻取）、MV_SALES_DAILY（日报）、MV_SALES_MONTHLY（月报/同比环比）

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 时间(日/月/年)、部门、业务员、品牌、大类、品类 |
| **核心指标** | 销售数量、销售流水、销售成本、毛利、毛利率 |
| **对比分析** | 同比增长率、环比增长率 |
| **更新频率** | 每日 |
| **数据来源** | MV_SALES_DAILY / MV_SALES_MONTHLY |

**指标公式**:
```sql
-- 毛利率（MV中不含，查询层计算）
ROUND(毛利 / NULLIF(销售流水, 0) * 100, 2) AS 毛利率

-- 同比增长率（基于MV_SALES_MONTHLY + LAG窗口函数）
ROUND((当月销售流水 - LAG(销售流水, 12)) / NULLIF(LAG(销售流水, 12), 0) * 100, 2) AS 同比增长率

-- 环比增长率
ROUND((当月销售流水 - LAG(销售流水, 1)) / NULLIF(LAG(销售流水, 1), 0) * 100, 2) AS 环比增长率
```

> **说明**: 原需求中的"目标完成率"已移除。当前系统无销售目标/预算数据表，如需此功能需先建立目标数据表（建议字段：年月、部门主键、业务员主键、目标流水、目标毛利）。

### 1.2 客户价值分析 ⭐⭐⭐
**业务价值**: 识别核心客户，优化客户结构
**MV依赖**: MV_SALES_DETAIL（销售明细）、MV_CUSTOMER_MASTER（客户属性）

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 时间(月/年)、客户、品牌、品类、地区(省份/城市) |
| **核心指标** | 销售流水、销售数量、订单数、毛利、客户贡献度、客户集中度 |
| **排名分析** | 客户销售额TOP N、品类销售占比 |
| **更新频率** | 每月 |
| **数据来源** | MV_SALES_DETAIL GROUP BY 客户主键 |

**指标公式**:
```sql
-- 客户贡献度 = 该客户销售流水 / 全部销售流水
ROUND(SUM(销售流水) / NULLIF(SUM(SUM(销售流水)) OVER (), 0) * 100, 2) AS 客户贡献度

-- 客户集中度 = 前N名客户销售额占比（CR5/CR10）
-- 用 ROW_NUMBER() + SUM() OVER() 实现帕累托分析

-- 订单数
COUNT(DISTINCT 销售结算主键) AS 订单数
```

### 1.3 客户活跃度监控 ⭐⭐
**业务价值**: 预警客户流失，提升客户留存
**MV依赖**: MV_CUSTOMER_ACTIVITY

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 客户、业务员、部门、活跃状态 |
| **核心指标** | 近30天订货天数、近30天订货金额、近90天订货天数、近90天订货金额、最后订货日期、距今天数、历史订单数、历史订货总金额 |
| **活跃分类** | 活跃(≤30天)、一般(31-60天)、沉睡(61-90天)、流失(>90天) |
| **预警规则** | 活跃状态='沉睡'或'流失'的客户列入预警名单 |
| **更新频率** | 每日（MV每日刷新） |
| **数据来源** | MV_CUSTOMER_ACTIVITY（基于SO_SALEINVOICE近一年数据） |

> **说明**: MV中活跃状态基于 CURRENT_DATE() 计算，每日刷新时更新，精度T+1。

### 1.4 新客户开发统计 ⭐⭐
**业务价值**: 评估市场拓展效果
**MV依赖**: MV_CUSTOMER_MASTER（创建日期）、MV_SALES_DETAIL（新客销售额）

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 时间(月/年)、业务员、部门、地区 |
| **核心指标** | 新客户数、新客销售额、新客占总销售额比例 |
| **对比分析** | 业务员新客开发数量排名 |
| **更新频率** | 每月 |
| **数据来源** | MV_CUSTOMER_MASTER (创建日期) + MV_SALES_DETAIL (销售数据) |

**新客户定义与计算逻辑**:
```sql
-- 新客户：统计周期内在MV_CUSTOMER_MASTER中创建日期在指定范围的客户
WHERE 创建日期 >= '2026-01-01' AND 创建日期 < '2026-02-01'

-- 新客销售额：该新客户在MV_SALES_DETAIL中的销售流水合计
-- 新客占比 = 新客销售额 / 全部销售额
```

> **说明**: 原需求中的"客户转化率"已移除——"转化"概念需要线索/商机管理数据，NC ERP不提供此类数据。如需跟踪首单到复购转化，可通过 MV_CUSTOMER_ACTIVITY 的历史订单数判断。

### 1.5 业务员绩效分析 ⭐⭐⭐
**业务价值**: 评估业务员表现，优化激励机制
**MV依赖**: MV_SALES_DETAIL（业绩数据）、MV_CUSTOMER_MASTER（管理客户数）

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 时间(月/年)、业务员、部门、品牌、品类 |
| **核心指标** | 管理客户数（活跃）、销售流水、毛利、毛利率、新客户数、客户数 |
| **排名分析** | 业务员业绩排名、同比增长排名 |
| **更新频率** | 每月 |
| **数据来源** | MV_SALES_DETAIL + MV_CUSTOMER_MASTER |

**指标公式**:
```sql
-- 管理客户数（活跃）：当期有销售记录的去重客户数
COUNT(DISTINCT 客户主键) AS 活跃客户数

-- 管理客户总数：MV_CUSTOMER_MASTER 中该业务员名下客户
SELECT COUNT(*) FROM MV_CUSTOMER_MASTER WHERE 业务员主键 = ? AND 是否封存 = 'N'
```

### 1.6 品类销售分析 ⭐⭐
**业务价值**: 优化产品组合，聚焦优势品类
**MV依赖**: MV_SALES_DETAIL、MV_SALES_MONTHLY

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 时间(月/年)、品牌、大类、品类、小类 |
| **核心指标** | 销售流水、销售数量、毛利、品类占比、同比增长率 |
| **分类方法** | ABC分类（A类=累计占比前80%、B类=80%-95%、C类=剩余） |
| **更新频率** | 每月 |
| **数据来源** | MV_SALES_DETAIL GROUP BY 品牌/大类/品类 |

**指标公式**:
```sql
-- 品类占比 = 该品类销售流水 / 总销售流水（公司内部占比）
ROUND(SUM(销售流水) / NULLIF(SUM(SUM(销售流水)) OVER (), 0) * 100, 2) AS 品类占比

-- ABC分类：按品类销售额降序排列，计算累计占比
SUM(品类占比) OVER (ORDER BY 销售流水 DESC) AS 累计占比
-- 累计占比 <= 80% → A类; <= 95% → B类; 其余 → C类
```

> **说明**: 原需求中的"市场份额"已改为"品类占比"。市场份额需外部行业数据，NC系统无法提供；品类占比反映公司内部产品结构，更有实操价值。

## 二、库存管理 (4个报表)

### 2.1 库存监控看板 ⭐⭐⭐
**业务价值**: 实时掌握库存状况，优化资金占用
**MV依赖**: MV_INVENTORY_SNAPSHOT

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 仓库、商品、品牌、大类、品类 |
| **核心指标** | 结存数量、结存单价、库存金额、库存占比 |
| **预警规则** | 零库存预警（结存数量=0 但近90天有销售）、超储预警（周转天数>90天） |
| **更新频率** | 每日 |
| **数据来源** | MV_INVENTORY_SNAPSHOT |

**指标公式**:
```sql
-- 库存占比
ROUND(库存金额 / NULLIF(SUM(库存金额) OVER (), 0) * 100, 2) AS 库存占比
```

> **说明**: 超储预警的"周转天数>90天"需结合 MV_SALES_DETAIL 的近期销售成本计算，公式见2.2。

### 2.2 库存周转分析 ⭐⭐⭐
**业务价值**: 识别呆滞库存，提高周转效率
**MV依赖**: MV_INVENTORY_SNAPSHOT（库存）、MV_SALES_DETAIL（销售成本）

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 时间(月/季)、仓库、商品、品牌、品类 |
| **核心指标** | 平均库存金额、销售成本合计、周转率、周转天数、呆滞库存金额 |
| **预警规则** | 周转天数>90天标记为滞销、180天无销售标记为呆滞 |
| **更新频率** | 每周 |
| **数据来源** | MV_INVENTORY_SNAPSHOT + MV_SALES_DETAIL |

**指标公式**:
```sql
-- 库存周转率 = 期间销售成本 / 平均库存金额
ROUND(SUM(销售成本) / NULLIF(AVG(库存金额), 0), 2) AS 周转率

-- 库存周转天数 = 期间天数 / 周转率（或 平均库存金额 * 期间天数 / 销售成本）
ROUND(期间天数 * AVG(库存金额) / NULLIF(SUM(销售成本), 0), 0) AS 周转天数

-- 呆滞库存：180天内在MV_SALES_DETAIL中无该商品销售记录的库存金额合计
```

### 2.3 出入库统计 ⭐⭐
**业务价值**: 监控货物流转，平衡出入库
**MV依赖**: 无（直接查询基表）

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 时间(日/月)、仓库、单据类型(入库/出库/盘点)、商品、品类 |
| **核心指标** | 入库单量、入库数量、入库金额、出库单量、出库数量、出库金额 |
| **平衡分析** | 出入库数量差异、日均出入库量趋势 |
| **更新频率** | 每日 |
| **数据来源** | IC_GENERAL_H + IC_GENERAL_B + BD_BILLTYPE + BD_INVBASDOC |

**查询模式**:
```sql
FROM IC_GENERAL_H h
    INNER JOIN IC_GENERAL_B b ON h.cgeneralhid = b.cgeneralhid AND b.dr = 0
    INNER JOIN BD_INVBASDOC inv ON b.cinvbasid = inv.pk_invbasdoc AND inv.dr = 0
    INNER JOIN BD_BILLTYPE bt ON h.cbilltypecode = bt.pk_billtypecode AND bt.dr = 0
    LEFT JOIN BD_STORDOC stor ON h.cwarehouseid = stor.pk_stordoc AND stor.dr = 0
WHERE h.pk_corp = '1006' AND h.dr = 0 AND h.fbillflag >= 3
```

> **待定**: 如查询频率高、数据量大，建议新建 MV_IO_SUMMARY（出入库汇总MV）。

### 2.4 调拨在途监控 ⭐⭐
**业务价值**: 优化仓间调配，减少在途时间
**MV依赖**: 无（直接查询基表）

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 调出仓、调入仓、商品、调拨路线(调出仓→调入仓) |
| **核心指标** | 在途数量、在途金额、在途天数 |
| **预警规则** | 超期在途标准：南京>4天、西安>5天、成都>10天、沈阳>11天 |
| **更新频率** | 每日 |
| **数据来源** | TO_BILL + TO_BILL_B + BD_STORDOC + BD_INVBASDOC |

**指标公式**:
```sql
-- 在途天数 = 当前日期 - 审批日期（查询层实时计算）
DATEDIFF(CURRENT_DATE(), CAST(zb.dauditdate AS DATE)) AS 在途天数

-- 在途条件：已审批(fstatus=5) 且 未关闭(fstatus!=7)
WHERE zb.fstatus = 5
```

**调拨路线定义**: 调出仓名称 + ' → ' + 调入仓名称，用于按路线分组统计平均在途时长。

## 三、财务管理 (4个报表)

### 3.1 应收账龄分析 ⭐⭐⭐
**业务价值**: 加强资金管理，降低坏账风险
**MV依赖**: MV_RECEIVABLE_SUMMARY

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 客户、业务员、部门、业务类型 |
| **核心指标** | 应收金额、未结算金额、账期天数、预计结算日期、超期天数 |
| **账龄分类** | 未到期、1-3天、4-10天、11-30天、31-90天、90天以上 |
| **预警规则** | 超期未结算金额 > 0 的客户列入预警 |
| **更新频率** | 每日 |
| **数据来源** | MV_RECEIVABLE_SUMMARY |

**指标公式**:
```sql
-- 超期天数（MV中不含，查询层实时计算）
DATEDIFF(CURRENT_DATE(), 预计结算日期) AS 超期天数

-- 账龄分类
CASE
    WHEN DATEDIFF(CURRENT_DATE(), 预计结算日期) <= 0 THEN '未到期'
    WHEN DATEDIFF(CURRENT_DATE(), 预计结算日期) <= 3 THEN '1-3天'
    WHEN DATEDIFF(CURRENT_DATE(), 预计结算日期) <= 10 THEN '4-10天'
    WHEN DATEDIFF(CURRENT_DATE(), 预计结算日期) <= 30 THEN '11-30天'
    WHEN DATEDIFF(CURRENT_DATE(), 预计结算日期) <= 90 THEN '31-90天'
    ELSE '90天以上'
END AS 账龄分类
```

### 3.2 客户信用管理 ⭐⭐⭐
**业务价值**: 控制信用风险，优化授信策略
**MV依赖**: MV_CUSTOMER_MASTER（信用额度）+ ARAP_DJFB（已用额度）

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 客户、业务员、部门 |
| **核心指标** | 信用额度、临时信用额度、总信用额度、已用额度、可用额度、额度使用率 |
| **预警规则** | 额度使用率>80%、可用额度<0（超限） |
| **更新频率** | 每日（T+1） |
| **数据来源** | MV_CUSTOMER_MASTER (信用额度) + ARAP_DJFB (未核销应收=已用额度) |

**指标公式**:
```sql
-- 已用额度 = ARAP_DJFB中该客户未核销的应收金额合计
SELECT SUM(fb.bbje) AS 已用额度
FROM ARAP_DJFB fb
WHERE fb.pk_corp = '1006' AND fb.dr = 0
    AND fb.djdl = 'ys' AND fb.isverifyfinished = 'N'
    AND fb.hbbm = ? -- 客户pk_cubasdoc

-- 可用额度 = 总信用额度 - 已用额度
-- 额度使用率 = ROUND(已用额度 / NULLIF(总信用额度, 0) * 100, 2)
```

> **说明**: 原需求更新频率为"实时"，已调整为"每日(T+1)"。MV架构为T+1刷新，无法实现实时。如需更高实时性，此报表可改用逻辑视图(VIEW)直接查询基表。

### 3.3 毛利分析 ⭐⭐⭐
**业务价值**: 评估产品盈利能力，优化产品结构
**MV依赖**: MV_SALES_DETAIL

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 时间(月/年)、部门、品牌、大类、品类、**商品** |
| **核心指标** | 销售流水、销售成本、毛利、毛利率 |
| **分析重点** | 月度毛利趋势、品类毛利对比、高低毛利商品TOP N |
| **更新频率** | 每月 |
| **数据来源** | MV_SALES_DETAIL GROUP BY 商品维度 |

> **与1.1的区别**: 1.1侧重**组织维度**(部门/业务员)的业绩达成，本报表侧重**商品维度**(品牌/品类/具体商品)的盈利能力分析，可下钻到单品级别。

### 3.4 经营日报 ⭐⭐⭐
**业务价值**: 全面掌握经营状况，快速决策
**MV依赖**: MV_SALES_DAILY + MV_INVENTORY_SNAPSHOT + MV_RECEIVABLE_SUMMARY

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 日期、部门 |
| **核心指标** | 当日销售流水、当日毛利、当月累计销售流水、当月累计毛利、库存总金额、未结算应收总额 |
| **对比分析** | 日环比（与昨日对比）、月累计同比（与去年同期累计对比） |
| **更新频率** | 每日 |
| **数据来源** | MV_SALES_DAILY (销售) + MV_INVENTORY_SNAPSHOT (库存) + MV_RECEIVABLE_SUMMARY (应收) |

**指标组成**:
```sql
-- 销售板块（来自 MV_SALES_DAILY）
SUM(销售流水) AS 当日销售流水,
SUM(毛利) AS 当日毛利

-- 库存板块（来自 MV_INVENTORY_SNAPSHOT）
SUM(库存金额) AS 库存总金额

-- 应收板块（来自 MV_RECEIVABLE_SUMMARY）
SUM(CASE WHEN 未结算金额 > 0 THEN 未结算金额 ELSE 0 END) AS 未结算应收总额
```

> **说明**: 原需求中的"采购金额"和"目标完成率"已移除。采购金额需 IC_GENERAL_H 入库数据（无对应MV），如需可后续补充。目标完成率需建立目标数据表。

## 四、采购管理 (3个报表)

### 4.1 采购入库统计 ⭐⭐
**业务价值**: 掌握采购到货情况，优化库存补货策略
**MV依赖**: 无（直接查询基表）

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 时间(日/月)、供应商、商品、品类、仓库 |
| **核心指标** | 入库单量、入库数量、入库金额 |
| **对比分析** | 月度入库趋势、供应商入库金额排名 |
| **更新频率** | 每周 |
| **数据来源** | IC_GENERAL_H + IC_GENERAL_B（筛选入库类型单据） |

**查询条件**:
```sql
-- 入库单据类型筛选：通过 BD_BILLTYPE 关联区分入库/出库
WHERE h.pk_corp = '1006' AND h.dr = 0 AND h.fbillflag >= 3
    AND bt.billtypename LIKE '%入库%'  -- 或根据具体单据类型编码筛选
```

> **说明**: 原标题"采购业绩统计"改为"采购入库统计"，更准确反映数据来源——IC_GENERAL_H 记录的是出入库行为而非采购订单。"采购与销售匹配度"因缺乏采购订单数据暂不可实现。

### 4.2 供应商分析 ⭐
**业务价值**: 了解供应商采购结构
**MV依赖**: MV_PRODUCT_MASTER（主供应商信息）

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 供应商、商品、品类 |
| **核心指标** | 入库金额、入库数量、供应商品数量 |
| **更新频率** | 每季度 |
| **数据来源** | IC_GENERAL_H + IC_GENERAL_B + MV_PRODUCT_MASTER |

> **说明**: 原"供应商绩效评估"已简化为"供应商分析"。原需求中的"交货及时率"需要采购订单的预计到货日期（NC系统未确认此字段是否已同步）、"质量合格率"需要质检数据（NC系统未部署质量管理模块）。当前仅基于入库数据进行采购金额和品类分析。

### 4.3 采购在途监控 ⭐
**业务价值**: 提高采购透明度，优化库存计划
**MV依赖**: 无

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 供应商、商品、采购单号 |
| **核心指标** | 在途数量、在途金额 |
| **预警规则** | 延期到货预警 |
| **更新频率** | 每日 |
| **数据来源** | 待确认 |

> **⚠ 待确认**: 需确认NC系统的采购订单表（PO_ORDER/PO_ORDER_B）是否已同步到StarRocks。如已同步，补充具体表名和关联方式；如未同步，此报表暂不可实现，建议在数据同步计划中优先纳入。

## 五、特殊业务 (3个报表)

### 5.1 电商业务监控 ⭐⭐
**业务价值**: 优化线上运营，提升电商业绩
**MV依赖**: 无（直接查询基表）

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 电商平台、商品、仓库、时间(日/月) |
| **核心指标** | 销售数量、销售金额、订单量 |
| **预警规则** | 缺货预警（库存<安全库存）、滞销预警 |
| **更新频率** | 每日 |
| **数据来源** | SO_SALEINVOICE + SO_SALEINVOICE_B（按业务类型筛选电商单据） |

> **⚠ 待确认**: 需明确电商平台的区分方式。可能的识别路径：

### 5.2 放货业务管理 ⭐⭐
**业务价值**: 控制放货风险，加快资金回笼
**MV依赖**: MV_RECEIVABLE_SUMMARY（通过业务类型筛选放货单据）

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 客户、业务员、商品、时间 |
| **核心指标** | 应收金额、未结算金额、超期天数、结算周期 |
| **预警规则** | 超期未结算>30天的放货单据 |
| **更新频率** | 每周 |
| **数据来源** | MV_RECEIVABLE_SUMMARY WHERE 业务类型 = '放货销售' |

**指标公式**:
```sql
-- 超期天数（查询层实时计算）
DATEDIFF(CURRENT_DATE(), 预计结算日期) AS 超期天数

-- 结算周期 = 实际结算日期 - 单据日期（需已结算记录）
```

### 5.3 退货分析 ⭐⭐
**业务价值**: 降低退货率，提升客户满意度
**MV依赖**: 无（直接查询基表）

| 报表项 | 内容 |
|--------|------|
| **分析维度** | 时间(月/年)、客户、商品、品牌 |
| **核心指标** | 退货数量、退货金额、退货率 |
| **分析重点** | 高退货商品识别、退货趋势、客户退货排名 |
| **更新频率** | 每月 |
| **数据来源** | SO_SALEINVOICE + SO_SALEINVOICE_B |

**指标公式**:
```sql
-- 退货单识别
WHERE ib.coriginalbillcode LIKE 'TH%' OR ib.coriginalbillcode LIKE '3U%'

-- 退货率 = 退货金额 / 同期销售金额（来自MV_SALES_MONTHLY）
ROUND(退货金额 / NULLIF(同期销售流水, 0) * 100, 2) AS 退货率
```

> **说明**: 原需求中的"退货原因"维度已移除。NC系统的销售发票/退货单中没有标准的退货原因字段，如需此维度需在NC系统中增加自定义字段并同步。

### 6.4 技术实现要点

**MV查询层计算字段**（以下字段不在MV中，需报表SQL实时计算）:
| 计算字段 | 公式 | 涉及报表 |
|---------|------|---------|
| 毛利率 | `ROUND(毛利/NULLIF(销售流水,0)*100, 2)` | 1.1, 1.5, 1.6, 3.3 |
| 超期天数 | `DATEDIFF(CURRENT_DATE(), 预计结算日期)` | 3.1, 5.2 |
| 同比增长率 | `ROUND((本期-去年同期)/NULLIF(去年同期,0)*100, 2)` | 1.1, 1.6 |
| 环比增长率 | `ROUND((本期-上期)/NULLIF(上期,0)*100, 2)` | 1.1 |
| 周转天数 | `ROUND(天数*AVG(库存金额)/NULLIF(SUM(销售成本),0), 0)` | 2.2 |

**报表交互要求**:
- 支持时间、部门、品牌、客户等维度的下拉筛选
- 支持从汇总到明细的钻取（月→日→明细）
- 异常数据（超期/超限/呆滞）高亮显示
- 支持Excel导出

**预警机制**:
| 预警类型 | 触发条件 | 涉及报表 |
|---------|---------|---------|
| 超期应收 | 超期天数 > 0 且 未结算金额 > 0 | 3.1 |
| 信用超限 | 已用额度 > 总信用额度 | 3.2 |
| 客户流失 | 活跃状态 = '沉睡' 或 '流失' | 1.3 |
| 库存呆滞 | 周转天数 > 90 或 180天无销售 | 2.1, 2.2 |
| 调拨超期 | 在途天数超过路线标准 | 2.4 |
| 放货超期 | 超期天数 > 30 且 未结算金额 > 0 | 5.2 |
