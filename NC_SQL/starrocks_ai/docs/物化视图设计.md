# 物化视图设计方案

> 基于 StarRocks 3.5 异步物化视图官方文档与BI报表需求的设计方案  
> 官方文档：https://docs.starrocks.io/zh/docs/3.5/category/asynchronous-materialized-views/

---

## 一、StarRocks 异步物化视图核心概念

### 1.1 什么是异步物化视图

StarRocks 异步物化视图（Async MV）是预计算并持久化存储的查询结果集。它可以：
- **加速复杂查询**：预先完成多表JOIN和聚合，避免重复计算
- **透明查询改写**：优化器可自动将查询改写为命中物化视图，无需修改SQL
- **数据建模**：替代传统ETL pipeline，用SQL声明式建模

### 1.2 适用场景判断

```
是否需要物化视图？
├─ 多个报表复用同一组多表JOIN（3张以上表关联） → 创建明细宽表MV
├─ 查询频率高 + 数据量大（>50万行）          → 创建聚合MV
├─ 需要复杂计算（如流水/毛利/账龄）          → 在MV中预计算
├─ 数据湖加速                                → 创建External Catalog MV
└─ 简单单表查询 / 实时性要求极高              → 不适合MV
```

### 1.3 视图 vs 物化视图 选择

| 特性 | 逻辑视图 (VIEW) | 异步物化视图 (MV) |
|------|-----------------|-------------------|
| 存储 | 不存储数据，无存储开销 | 存储预计算结果，有额外存储 |
| 更新 | 无更新开销 | 基表变更时需刷新 |
| 性能 | 不做预计算，无性能收益 | 预计算结果，加速查询 |
| 实时性 | 查询时返回最新数据 | 可能存在数据延迟（取决于刷新频率） |
| 适用场景 | 业务语义封装、SQL简化 | 数据建模、透明加速、报表查询 |

> **项目决策**：本项目报表场景允许T+1数据延迟，优先使用物化视图加速。对于实时大屏等场景，使用逻辑视图 + 原始表查询。

## 二、DDL语法规范（基于StarRocks 3.5）

### 2.1 完整DDL模板

```sql
CREATE MATERIALIZED VIEW [IF NOT EXISTS] {mv_name}
-- 注释
COMMENT '{中文描述}'
-- 分区策略（大数据量MV推荐）
[PARTITION BY 
    {base_column}                              -- 等比例对齐
    | date_trunc('{granularity}', {column})    -- 时间粒度上卷
    | time_slice({column}, INTERVAL {n} {unit}) -- 自定义粒度
]
-- 分桶策略
DISTRIBUTED BY HASH({column}) [BUCKETS {n}]
-- 刷新策略
REFRESH 
    ASYNC                                      -- 基表变更自动刷新
    | ASYNC EVERY (INTERVAL {n} {unit})        -- 定时刷新
    | MANUAL                                   -- 仅手动刷新
-- 属性配置
PROPERTIES (
    -- 刷新控制
    "partition_refresh_number" = "1",           -- 每次刷新分区数（v3.3+默认1）
    "auto_refresh_partitions_limit" = "-1",     -- 自动刷新分区上限（-1=不限）
    "excluded_trigger_tables" = "",             -- 不触发刷新的基表
    -- 数据保留
    "partition_ttl" = "90 day",                 -- 分区TTL（推荐，v3.1.4+）
    -- "partition_ttl_number" = "90",           -- 旧版：保留最近N个分区
    -- 性能
    "enable_spill" = "true",                    -- 大MV启用落盘防OOM（v3.1+）
    "resource_group" = "mv_refresh_rg",         -- 资源组隔离（v3.1+）
    -- 查询改写
    "mv_rewrite_staleness_second" = "86400",    -- 允许1天staleness的查询改写
    "enable_query_rewrite" = "true"             -- 是否可用于查询改写（v3.3+）
)
AS
{query_statement};
```

### 2.2 三种刷新策略对比

| 策略 | 语法 | 触发方式 | 适用场景 |
|------|------|---------|---------|
| **自动刷新** | `REFRESH ASYNC` | 基表数据变更时自动触发 | StarRocks内表、数据变更频率可控 |
| **定时刷新** | `REFRESH ASYNC EVERY(INTERVAL 1 DAY)` | 按固定间隔定时触发 | 外表、需要固定刷新窗口 |
| **手动刷新** | `REFRESH MANUAL` | 仅手动执行REFRESH | 外部调度、特殊控制需求 |

> **项目选择**：由于数据来自用友NC（外部数据源同步到StarRocks），数据变更时间不固定，推荐使用 **定时刷新**（`REFRESH ASYNC EVERY`）为主，配合手动刷新。

### 2.3 分区策略详解

#### 为什么要分区？

- **增量刷新**：只刷新有变化的分区，而不是全量重建
- **TTL管理**：自动淘汰过期分区，控制存储成本
- **透明改写**：过期分区的查询自动退回基表，保证数据一致性
- **构建控制**：分批构建，避免一次性消耗过多资源

#### 分区策略选择

| 方式 | 语法 | 效果 | 适用版本 |
|------|------|------|---------|
| 等比例对齐 | `PARTITION BY datekey` | MV分区与基表一一对应 | v2.5+ |
| 时间上卷 | `PARTITION BY date_trunc('month', datekey)` | MV按月分区，基表按日 | v2.5+ |
| 自定义粒度 | `PARTITION BY date_trunc('day', time_slice(datekey, INTERVAL 5 HOUR))` | 自定义时间周期 | v3.2+ |

> **注意**：分区物化视图只能基于已分区的基表创建。如果基表未分区，MV也不能分区。

### 2.4 关键PROPERTIES说明

| 属性 | 默认值 | 说明 | 版本 |
|------|--------|------|------|
| `partition_refresh_number` | 1 (v3.3+) | 每次刷新的分区数，-1=不限。避免一次刷新太多分区占资源 | v2.5+ |
| `auto_refresh_partitions_limit` | -1 | 每次自动刷新的分区上限 | v2.5+ |
| `excluded_trigger_tables` | "" | 列出的表数据变更不触发MV刷新。适用于维度表频繁变更场景 | v2.5+ |
| `partition_ttl` | 无 | 分区数据保留时间，如`"90 day"`。推荐替代partition_ttl_number | v3.1.4+ |
| `partition_ttl_number` | -1 | 保留最近N个分区，-1=全部保留 | v2.5+ |
| `enable_spill` | false | 构建时中间结果落盘，大MV必须开启防OOM | v3.1+ |
| `resource_group` | default | 指定MV刷新的资源组，实现资源隔离 | v3.1+ |
| `mv_rewrite_staleness_second` | 0 | 允许的数据陈旧秒数（用于查询改写） | v3.1+ |
| `enable_query_rewrite` | true | 是否允许此MV用于查询改写 | v3.3+ |

### 3.1 视图总览

| 序号 | 视图名称 | 优先级 | 刷新策略 | 排序键 | 数据量级 | 服务报表数 |

| 2 | MV_INVENTORY_SNAPSHOT | ⭐⭐⭐ | DEFERRED ASYNC EVERY 1 DAY | 仓库主键, 商品主键 | 中(十万级) | 2 |
| 3 | MV_RECEIVABLE_SUMMARY | ⭐⭐⭐ | DEFERRED ASYNC EVERY 1 DAY | 单据日期, 客户主键 | 中(十万级) | 2 |
| 4 | MV_CUSTOMER_MASTER | ⭐⭐ | DEFERRED ASYNC EVERY 1 DAY | 客户编码 | 小(万级) | 多个 |
| 5 | MV_PRODUCT_MASTER | ⭐⭐ | DEFERRED ASYNC EVERY 1 DAY | 商品编码, 品牌 | 中(十万级) | 多个 |
| 6 | MV_SALES_DAILY | ⭐⭐ | DEFERRED ASYNC EVERY 1 DAY | 日期, 部门主键, 业务员主键 | 小(千级) | 2 |
| 7 | MV_SALES_MONTHLY | ⭐⭐ | DEFERRED ASYNC EVERY 1 DAY | 年月, 部门主键, 业务员主键 | 小(百级) | 2 |
| 8 | MV_CUSTOMER_ACTIVITY | ⭐ | DEFERRED ASYNC EVERY 1 DAY | 客户主键 | 小(万级) | 1 |

### 3.2 视图层级依赖

```
Layer 0 - 原始表（用友NC同步）
    SO_SQUAREDETAIL, IC_ONHANDNUM, SO_SQUARE, BD_CUMANDOC, ...

Layer 1 - 明细/主数据视图
    MV_SALES_DETAIL      ← SO_SQUAREDETAIL + 维度表
    MV_INVENTORY_SNAPSHOT ← IC_ONHANDNUM + 维度表 + 成本表
    MV_RECEIVABLE_SUMMARY ← SO_SQUARE + 维度表
    MV_CUSTOMER_MASTER    ← BD_CUMANDOC + 维度表 + 信用表
    MV_PRODUCT_MASTER     ← BD_INVBASDOC + BD_INVMANDOC

Layer 2 - 汇总视图
    MV_SALES_DAILY        ← 直接聚合原始表
    MV_SALES_MONTHLY      ← 直接聚合原始表
    MV_CUSTOMER_ACTIVITY  ← SO_SALEINVOICE + 维度表
```

> **刷新顺序**：Layer 1 先刷新 → Layer 2 后刷新。如果 Layer 2 依赖 Layer 1，StarRocks会自动管理级联刷新。

## 四、推荐DDL写法（对比改进）

### 4.1 改进对照（已完成）

| 改进项 | 改进前 | 改进后 | 原因 |
|--------|--------|--------|------|
| IF NOT EXISTS | 无 | 已添加 | 防止重复创建报错 |
| COMMENT | 无 | 已添加中文描述 | 可通过 SHOW MATERIALIZED VIEWS 查看用途 |
| REFRESH策略 | `REFRESH ASYNC` | `REFRESH DEFERRED ASYNC EVERY(INTERVAL 1 DAY)` | 定时刷新更可控；DEFERRED 避免创建时立即刷新 |
| enable_spill | 未设置 | `"true"` | 大MV构建时防OOM |
| mv_rewrite_staleness_second | 未设置 | `"86400"` | 允许1天staleness，启用查询改写加速 |
| excluded_trigger_tables | 未设置 | 已为事实表MV设置维度表排除 | 维度表变更不触发中间刷新 |
| CREATE INDEX | 独立DDL（语法无效） | 已改为 `ORDER BY` 排序键 | StarRocks MV不支持CREATE INDEX，ORDER BY是正确的索引方式 |
| CURRENT_DATE() | MV内部使用（MV02快照日期、MV03超期天数） | MV02已移除；MV03超期天数改为查询层计算 | MV中的CURRENT_DATE()在刷新时固定，之后立即过时 |
| BD_PAYTERM.pk_corp | MV03、MV04缺少 | 已补充 `pk_corp = '1006'` | BD_PAYTERM属于管理档案，需要pk_corp过滤 |
| IA_GENERALLEDGER月份 | MV02未过滤最新月份（可能行重复） | 已用MAX(caccountmonth)子查询修复 | 防止多月份数据导致库存行重复 |
| MV05 invcode过滤 | NOT IN + SUBSTR 冗余 | 统一为SUBSTR | NOT IN列表已被SUBSTR完全覆盖 |
| 毛利率 | MV06/07中4次重复公式 | 移至查询层计算 | 减少MV复杂度，毛利率=毛利/NULLIF(销售流水,0)*100 |
| MV07维度 | 含小类（与DAILY不一致） | 移除小类，与DAILY统一 | 月汇总粒度应≥日汇总粒度 |
| MV08指标名 | "历史订货总次数"（实为天数） | 改为COUNT(DISTINCT csaleid)=历史订单数 | 原指标COUNT(DISTINCT dbilldate)统计的是天数 |
| MV03 cbiztype | 硬编码过滤单一业务类型 | 移除，改为列输出+查询层过滤 | 提高MV通用性 |
| 刷新脚本 | 无WITH SYNC MODE，状态查询语法错误 | 已加SYNC MODE，改用SHOW MATERIALIZED VIEWS | ASYNC刷新不等待完成；information_schema.tables对MV无效 |

### 4.2 当前标准DDL模板

```sql
CREATE MATERIALIZED VIEW IF NOT EXISTS MV_SALES_DETAIL
COMMENT '销售明细宽表 - 支持所有销售分析报表'
DISTRIBUTED BY HASH(销售结算主键) BUCKETS 32
ORDER BY (制单日期, 部门主键, 客户主键)
REFRESH DEFERRED ASYNC EVERY (INTERVAL 1 DAY)
PROPERTIES (
    "enable_spill" = "true",
    "mv_rewrite_staleness_second" = "86400",
    "excluded_trigger_tables" = "BD_PSNDOC,BD_DEPTDOC,BD_CUBASDOC,BD_AREACL,BD_BUSITYPE"
)
AS
SELECT ... FROM SO_SQUAREDETAIL sq ...;
```

### 4.4 分区MV写法（如果基表已分区）

如果 `SO_SQUAREDETAIL` 表按日期分区，可以创建分区MV实现增量刷新：

```sql
CREATE MATERIALIZED VIEW IF NOT EXISTS MV_SALES_DETAIL
COMMENT '销售明细宽表 - 按月分区增量刷新'
PARTITION BY date_trunc('month', 制单日期)
DISTRIBUTED BY HASH(销售结算主键) BUCKETS 32
REFRESH DEFERRED ASYNC EVERY (INTERVAL 1 DAY)
PROPERTIES (
    "partition_refresh_number" = "1",
    "partition_ttl" = "730 day",
    "enable_spill" = "true",
    "mv_rewrite_staleness_second" = "86400"
)
AS
SELECT ... FROM SO_SQUAREDETAIL sq ...;
```

**分区MV的优势**：
- 每次只刷新有变化的月份分区，不需全量重建
- 通过 `partition_ttl` 自动保留最近2年数据
- 首次构建可分批执行，不会一次性占满资源

## 五、刷新策略详解

### 5.1 推荐刷新配置

**所有MV统一使用每日定时刷新**：
```sql
-- 全部MV: REFRESH DEFERRED ASYNC EVERY (INTERVAL 1 DAY)
-- MV_SALES_DETAIL       - 销售明细宽表
-- MV_INVENTORY_SNAPSHOT  - 库存快照表
-- MV_RECEIVABLE_SUMMARY  - 应收账款汇总表
-- MV_CUSTOMER_MASTER     - 客户主数据
-- MV_PRODUCT_MASTER      - 商品主数据
-- MV_SALES_DAILY         - 销售日汇总
-- MV_SALES_MONTHLY       - 销售月汇总
-- MV_CUSTOMER_ACTIVITY   - 客户活跃度（活跃状态需每日更新）
```

> **DEFERRED**：创建时不立即刷新，在首个调度时间点或手动触发时执行首次构建。

### 5.2 手动刷新命令

```sql
-- 普通刷新（跳过无变化的分区）
REFRESH MATERIALIZED VIEW MV_SALES_DETAIL;

-- 同步刷新（等待完成再返回）
REFRESH MATERIALIZED VIEW MV_SALES_DETAIL WITH SYNC MODE;

-- 强制全量刷新（忽略变化检测）
REFRESH MATERIALIZED VIEW MV_SALES_DETAIL FORCE WITH SYNC MODE;

-- 刷新指定分区范围
REFRESH MATERIALIZED VIEW MV_SALES_DETAIL
PARTITION START ("2026-01-01") END ("2026-02-01")
FORCE WITH SYNC MODE;
```

### 5.3 维度表变更不触发刷新

对于维度表（如人员、部门）频繁变更但不需要立即刷新的场景：

```sql
-- 创建MV时设置排除触发表
PROPERTIES (
    "excluded_trigger_tables" = "BD_PSNDOC,BD_DEPTDOC,BD_CUBASDOC"
)
```

这样人员/部门档案变更不会触发MV刷新，只在定时刷新时才更新。

## 六、监控与运维

### 6.1 查看MV状态

```sql
-- 查看所有物化视图基本信息
SHOW MATERIALIZED VIEWS;

-- 查看指定MV的详细信息
SHOW MATERIALIZED VIEWS WHERE NAME = 'MV_SALES_DETAIL';

-- 关注字段：
-- is_active: 是否激活（TRUE才能正常工作）
-- last_refresh_state: 上次刷新状态（SUCCESS/FAILED）
-- last_refresh_error_message: 失败原因
-- rows: 数据行数
```

### 6.2 查看刷新Task Run详情

```sql
-- 查看最近10次刷新记录
SELECT 
    TASK_NAME,
    CREATE_TIME,
    FINISH_TIME,
    STATE,
    `DATABASE`,
    DEFINITION
FROM information_schema.task_runs
WHERE TASK_NAME LIKE 'mv-%'
ORDER BY CREATE_TIME DESC
LIMIT 10;

-- 解析刷新详情（v3.4+）
SELECT 
    TASK_NAME,
    CREATE_TIME,
    STATE,
    get_json_string(EXTRA_MESSAGE, '$.refreshMode') AS 刷新模式,
    get_json_string(EXTRA_MESSAGE, '$.forceRefresh') AS 是否强制,
    get_json_string(EXTRA_MESSAGE, '$.mvPartitionsToRefresh') AS 刷新分区,
    get_json_string(EXTRA_MESSAGE, '$.partitionStart') AS 分区起始,
    get_json_string(EXTRA_MESSAGE, '$.partitionEnd') AS 分区结束
FROM information_schema.task_runs
WHERE TASK_NAME LIKE 'mv-%'
ORDER BY CREATE_TIME DESC
LIMIT 10;
```

### 6.3 计算刷新耗时

```sql
SELECT 
    TASK_NAME AS 任务名,
    CREATE_TIME AS 创建时间,
    FINISH_TIME AS 完成时间,
    STATE AS 状态,
    TIMESTAMPDIFF(SECOND, CREATE_TIME, FINISH_TIME) AS 总耗时秒
FROM information_schema.task_runs
WHERE TASK_NAME LIKE 'mv-%' 
    AND STATE = 'SUCCESS'
ORDER BY CREATE_TIME DESC
LIMIT 20;
```

### 6.5 故障排除

#### 问题1：MV变为Inactive状态

**原因**：基表发生Schema Change（如加列、改类型）导致MV失效。

```sql
-- 检查MV是否Active
SHOW MATERIALIZED VIEWS WHERE NAME = 'MV_SALES_DETAIL'\G

-- 手动修复：重新激活
ALTER MATERIALIZED VIEW MV_SALES_DETAIL ACTIVE;

-- 如果ACTIVE失败（SQL定义不兼容新Schema），需要重建
DROP MATERIALIZED VIEW IF EXISTS MV_SALES_DETAIL;
-- 修改DDL后重新创建
```

> **v3.1.4+自动修复**：StarRocks后台会尝试自动激活Inactive的MV，初始间隔30秒，指数递减直到60min后停止。

#### 问题2：刷新失败

```sql
-- 查看失败详情
SELECT 
    TASK_NAME, STATE, ERROR_MESSAGE,
    get_json_string(EXTRA_MESSAGE, '$.refreshMode') AS 模式
FROM information_schema.task_runs
WHERE STATE = 'FAILED'
ORDER BY CREATE_TIME DESC
LIMIT 5;

-- 常见原因及处理：
-- 1. OOM → 设置 "enable_spill" = "true"
-- 2. 超时 → 设置 PROPERTIES("session.query_timeout"="7200")
-- 3. 数据异常 → 检查基表数据，修复后强制刷新
```

#### 问题3：刷新耗时过长

```sql
-- 分析刷新了多少分区
SELECT 
    TASK_NAME,
    get_json_string(EXTRA_MESSAGE, '$.mvPartitionsToRefresh') AS 分区列表,
    get_json_int(EXTRA_MESSAGE, '$.adaptivePartitionRefreshNumber') AS 批次大小
FROM information_schema.task_runs
WHERE TASK_NAME LIKE 'mv-%'
ORDER BY CREATE_TIME DESC
LIMIT 5;

-- 优化建议：
-- 1. 降低 partition_refresh_number（每次刷新更少分区）
-- 2. 使用 excluded_trigger_tables 排除不重要的维度表变更
-- 3. 设置 partition_ttl 减少需要维护的分区数
```

#### 问题4：查询未命中MV（查询改写失败）

```sql
-- 诊断查询改写（v2.5.10+）
TRACE REWRITE SELECT ... FROM SO_SQUAREDETAIL ...;

-- 常见原因：
-- 1. MV处于Inactive状态
-- 2. MV数据过旧，超过 mv_rewrite_staleness_second
-- 3. 查询条件与MV不匹配
-- 4. enable_materialized_view_rewrite 被关闭
SET enable_materialized_view_rewrite = true;
```

## 七、高级特性与最佳实践

### 7.1 原子替换MV（v3.1+）

当需要修改MV定义（如增加字段）时，使用原子替换避免中断查询：

```sql
-- 1. 创建新版本MV
CREATE MATERIALIZED VIEW MV_SALES_DETAIL_V2
...
AS SELECT ..., new_column ...;

-- 2. 等待新MV刷新完成
REFRESH MATERIALIZED VIEW MV_SALES_DETAIL_V2 WITH SYNC MODE;

-- 3. 原子替换（交换名字，不中断服务）
ALTER MATERIALIZED VIEW MV_SALES_DETAIL SWAP WITH MV_SALES_DETAIL_V2;

-- 4. 删除旧MV
DROP MATERIALIZED VIEW MV_SALES_DETAIL_V2;
```

### 7.2 排序键优化（v3.1+）

在MV创建时指定排序键，加速常见查询模式的过滤：

```sql
CREATE MATERIALIZED VIEW MV_SALES_DETAIL
...
-- 排序键：日期最常用于范围查询
ORDER BY (日期, 部门主键, 业务员主键)
AS ...;
```

### 7.3 嵌套MV（多层建模）

可以基于MV创建MV，实现分层聚合：

```sql
-- Layer 1: 明细宽表
CREATE MATERIALIZED VIEW MV_SALES_DETAIL ... AS SELECT ... FROM SO_SQUAREDETAIL ...;

-- Layer 2: 基于Layer 1的月度汇总（更高效）
CREATE MATERIALIZED VIEW MV_SALES_MONTHLY ... AS SELECT ... FROM MV_SALES_DETAIL ...;
```

> **级联刷新**：当MV_SALES_DETAIL刷新完成后，MV_SALES_MONTHLY会自动触发刷新。

### 7.4 查询改写（透明加速）

StarRocks优化器可以自动将针对基表的查询改写为使用MV。支持的改写类型：

| 改写类型 | 说明 | 版本 |
|---------|------|------|
| 单表改写 | 单表查询自动命中MV | v2.5+ |
| Inner Join改写 | 多表JOIN查询命中MV | v2.5+ |
| 聚合改写 | SUM/COUNT等聚合命中MV | v2.5+ |
| UNION补偿改写 | MV只有部分分区有效时，自动UNION基表查询 | v2.5+ |
| View Delta改写 | 查询的表是MV的子集 | v2.5.4+ |
| 文本改写 | AST完全匹配 | v3.3+ |

**实际意义**：如果MV_SALES_DETAIL包含了销售毛利的所有字段和计算，那么直接查询 `SO_SQUAREDETAIL` 的报表SQL也可能自动改写为查询MV，无需修改SQL。

### 7.5 DEFERRED 延迟刷新

创建大表MV时，使用 `DEFERRED` 避免立即触发全量构建阻塞：

```sql
-- 创建后不立即刷新
CREATE MATERIALIZED VIEW MV_SALES_DETAIL
REFRESH DEFERRED ASYNC EVERY (INTERVAL 1 DAY)
AS ...;

-- 在合适的时间窗口手动触发首次刷新
REFRESH MATERIALIZED VIEW MV_SALES_DETAIL WITH SYNC MODE;
```

## 八、使用示例

### 8.1 销售业绩报表查询（使用MV）

```sql
SELECT 
    日期, 部门名称, 业务员姓名, 品牌,
    SUM(销售数量) AS 销售数量,
    ROUND(SUM(销售流水), 2) AS 销售流水,
    ROUND(SUM(销售成本), 2) AS 销售成本,
    ROUND(SUM(毛利), 2) AS 毛利
FROM MV_SALES_DETAIL
WHERE 日期 BETWEEN '2026-01-01' AND '2026-03-31'
    AND 是否赠品 = 'N'
GROUP BY 日期, 部门名称, 业务员姓名, 品牌
ORDER BY 日期 DESC;
```

### 8.2 库存监控查询

```sql
SELECT 
    仓库名称, 品牌, 大类,
    SUM(结存数量) AS 库存数量,
    ROUND(SUM(库存金额), 2) AS 库存金额
FROM MV_INVENTORY_SNAPSHOT
GROUP BY 仓库名称, 品牌, 大类
ORDER BY 库存金额 DESC;
```

### 8.3 应收账龄查询

```sql
-- 超期天数在查询层实时计算，避免MV中CURRENT_DATE()过时
SELECT 
    客户名称, 业务员姓名, 部门名称,
    DATEDIFF(CURRENT_DATE(), 预计结算日期) AS 超期天数,
    CASE 
        WHEN DATEDIFF(CURRENT_DATE(), 预计结算日期) <= 0 THEN '未到期'
        WHEN DATEDIFF(CURRENT_DATE(), 预计结算日期) <= 3 THEN '1-3天'
        WHEN DATEDIFF(CURRENT_DATE(), 预计结算日期) <= 10 THEN '4-10天'
        WHEN DATEDIFF(CURRENT_DATE(), 预计结算日期) <= 30 THEN '11-30天'
        WHEN DATEDIFF(CURRENT_DATE(), 预计结算日期) <= 90 THEN '31-90天'
        ELSE '90天以上'
    END AS 账龄分类,
    SUM(应收金额) AS 应收金额
FROM MV_RECEIVABLE_SUMMARY
WHERE 未结算金额 > 0
GROUP BY 客户名称, 业务员姓名, 部门名称,
    DATEDIFF(CURRENT_DATE(), 预计结算日期)
ORDER BY 应收金额 DESC;
```
